# Session insights

Что-то, что стоит запомнить из этой сессии. Не пересказ, а conclusions, которые будут полезны в следующий раз.

## CSS / анимации

### Shorthand `transition` в двух классах дерётся
Если на элементе одновременно `.A` и `.B`, и в обеих написан `transition: ...` shorthand, побеждает последний по cascade-порядку. Список перенесётся целиком, а не объединится. Симптом — пропавшие переходы для свойств, которые перечислены только в проигравшем классе (иконка телепортируется в центр, потому что `top/left/width/height` пропали из transition-листа).

Решение: либо все transition в одном правиле, либо в каждом — **полный** список свойств. Не доверять «по-property merge» — его нет.

### Single rAF не даёт браузеру отдельный layout/paint
Между добавлением класса с `opacity: 0 → 1` и следующего с `opacity: 1 → 0` нужен реальный отрисованный кадр в промежутке, иначе оба class-add'а резолвятся в одном style recalc и `opacity` сразу 0 (без transition).

Решение: **double-rAF** (rAF внутри rAF). Первый колбэк всё ещё в кадре до paint; второй — гарантированно после.

```js
requestAnimationFrame(() => {
  requestAnimationFrame(() => {
    el.classList.add('is-faded');  // теперь opacity транзишн от закоммиченной 1 → 0
  });
});
```

### PNG-фильтры `brightness(0) invert(1)` схлопывают непрозрачные пиксели
Все непрозрачные пиксели становятся одного цвета. Бренд-мар с белыми буквами «ОК» на оранжевой плитке → один белый квадрат. Не работает как «перекраска в белое». Если нужен только цвет букв — либо отдельный PNG с прозрачным фоном, либо MASK через CSS `mask-image`.

### Подложка контейнера светится сквозь прозрачные углы PNG
PNG-плитка с прозрачными углами + контейнер с background fill → через углы видно «второй» оттенок. Фикс: убирать фон у контейнера, тень тогда давать через `filter: drop-shadow` (она следует за alpha PNG, а не за прямоугольником).

## Проверка через реальный браузер

### Computed style + sessionStorage > скриншоты
Скриншот говорит «выглядит ок», а computed `opacity: 0` показывает, что иконка вообще не отрендерилась — её просто не было видно. Полезный паттерн: внутри `page.evaluate` собирать кадры через `requestAnimationFrame` и возвращать массив. ~16 ms точность без host round-trip.

### Перехват навигации `page.route` для наблюдения за оверлеями
`#launch` живёт только во время splash-анимации (потом `window.location.href` уносит на следующую страницу, и элемент исчезает). Чтобы сэмплить весь полёт:

```js
await page.route('**/lenta-light.html', r => r.abort('aborted'));
```

Тогда тап стартует анимацию, но навигация не происходит → элемент остаётся, можно мерить хоть до бесконечности.

### Screencast CDP для покадрового видео
`page.screenshot` блокирующий (~50 ms на кадр). Для покадровой анимации:
```js
const cdp = await page.context().newCDPSession(page);
await cdp.send('Page.startScreencast', { format: 'png', everyNthFrame: 1 });
cdp.on('Page.screencastFrame', e => frames.push({ t: e.metadata.timestamp, data: e.data }));
```
Кадр на каждый paint без подтормаживания.

## Архитектура

### sessionStorage для cross-page состояния
Прототип мульти-страничный (`start.html`, `lenta-light.html`, …). Любое состояние, которое должно переживать навигацию (стрим уведомлений, флаг «уже за локскрином»), кладётся в `sessionStorage`. Простые ключи (`ok_notif_idx`, `okstart_at_home`) проще, чем event-bus или synced-stores. Работает «насрать что происходит на экране».

### Декларативный component-API через `data-` атрибуты
Лучший интерфейс компонента для прототипа: положить `<div class="X" data-autostart data-mode="…">` + подключить JS, дальше JS сам подцепится по `[data-*]` на `DOMContentLoaded`. Не нужно ручных вызовов API — три тега и работает. Для динамики оставить программные методы (`OkNotifs.setMode`, `setItems`) поверх декларативных дефолтов.

### BEM-модификаторы для вариантов раскладки
`.notifs.__mode-lock` (flat list) vs `.notifs.__mode-heads-up` (iOS deck) — один компонент, два состояния. Логика выбирает стратегию по классу контейнера, CSS даёт раскладку. Можно переключать `setMode()`, можно ставить в HTML декларативно. Чем меньше булевых пропсов в JS — тем легче использовать.

### Грэп существующее ПРЕЖДЕ чем писать новое
Я сделал `back-button.js`, не посмотрев, что `screen-transition.js` уже централизованно ловит `.nav-bar__back`. Перед написанием любого «полезного утилитарного компонента» — `grep -rn 'cssClass\|aria-label' components/`. Сэкономит pull request, ревью и стыд.

## Subagent в Claude Code

### Поле `tools` лучше не задавать, если используешь env-specific инструменты
`SendUserFile` нет в официальном tools-reference (он от «Claude Code on the web»). Если перечислить его в `tools:` — могут проигнорировать список, и агент останется без инструмента. Лучше **омитить** поле `tools` целиком, тогда subagent наследует все инструменты родительской сессии, включая env-specific.

### `memory: project` для self-learning
Даёт директорию `.claude/agent-memory/<agent>/`, в которой первые 200 строк / 25 KB `MEMORY.md` **автоматически** инжектятся в системный промт. Достаточно сказать в промте «обнови MEMORY.md перед возвратом, прочитай перед началом». Файл коммитится в git → знания переживают сессии и шарятся в команде.

### Subagent не видит SendUserFile / AskUserQuestion
В стандартной спеке: `Agent, AskUserQuestion, EnterPlanMode, ExitPlanMode, ScheduleWakeup, WaitForMcpServers` — не доступны subagent'ам. `SendUserFile` фактически тоже может быть недоступен. Fallback: `Read` на PNG из `/tmp` — изображение попадает в транскрипт, родительский агент его видит, может переслать.

### Subagent подхватывается только на старте сессии
Любые изменения в `.claude/agents/*.md` требуют рестарта сессии. Через UI `/agents` — без рестарта, но это интерактивный путь. Для git-flow проще закоммитить и сказать «в новой сессии заработает».

## Процесс

### Long-running ветка ловит конфликты на каждом мердже
Параллельная работа на main → каждый мой PR конфликтует по тем же файлам. Тратится время на ресинхронизацию. Когда ясно, что будет 5+ итераций — лучше **один** длинный PR (или серия мелких + регулярный merge main в свою ветку), чем 15 PR с конфликтом каждый раз.

### Verification != прохождение тестов
Юнит-тест говорит «функция вернула 42». Verification говорит «иконка пролетела от A до B за 500ms, opacity ушёл 1→0, splashLogo проявился из блюра». Это про реальное поведение в браузере. Гонять тесты как доказательство — это re-running CI, не verification.

### «Запушено» != «работает»
Несколько раз в этой сессии я писал «готово» на основе того, что код выглядит правильно — а потом Playwright показывал, что иконка opacity=0 на всех кадрах. Дисциплина: не отчитываться о готовности фичи, пока не подёргал её в браузере. Если не можешь подёргать — скажи это явно, не подменяй фразами «должно работать».
