# Verify UI (browser scenarios)

Запускает агент **screenshot-testing** (Playwright + Chromium, локальный сервер из корня
репо) с готовым промптом под типовой сценарий. Снимает кадры + computed styles/DOM/
sessionStorage, возвращает вердикт.

## Как пользоваться
1. Определи **сценарий** (из шаблонов ниже) и параметры: какие страницы/селекторы.
2. Запусти агент `screenshot-testing` (Agent tool, subagent_type: screenshot-testing),
   подставив параметры в шаблон. Сервер агент поднимает сам из **корня** репо (относительные
   пути к index.css/assets/components должны резолвиться).
3. Перескажи пользователю вердикт + что показали скриншоты; если FAIL — почини и перезапусти.

## Общие правила (грабли из практики)
- Стартовать сценарий «назад» нельзя с прямого `goto X.html` — `history.back()` уведёт на
  `about:blank`. Сначала forward (A→B), потом back (B→A).
- Игнорировать посторонний консольный шум: `CERT|404|net::|jsdelivr|lottie|favicon`.
- Анимации мерить через `getAnimations()` + `animationName` на нужном элементе/псевдоэлементе;
  для VT — `::view-transition-old/new(root)`; ловить середину перехода (замедлить/семплить rAF).
- Если cross-document View Transitions не активируется на localhost — отметить и проверить
  хотя бы логику (классы/флаги/навигацию), а не только визуал.

## Шаблоны промптов

### A. Системный переход назад/вперёд (screen-transition)
> Проверь системный переход в ds-design (рабочая копия), Chromium, сервер из корня.
> FORWARD: открой `<A>.html`, тапни `<селектор-перехода-вперёд>` → должен открыться `<B>.html`,
> экран въезжает справа (`::view-transition-new(root)` = `screen-in-right`, old = `screen-out-left`),
> `html.nav-back` НЕ стоит. BACK: на `<B>.html` тапни `.nav-bar__back` → `history.back()` → `<A>.html`,
> во время перехода `document.documentElement.classList.contains('nav-back') === true`,
> new=`screen-in-left`, old=`screen-out-right` (текущий уезжает вправо поверх). Верни таблицу
> (nav-back / keyframes old,new / URL), скриншоты середины обоих переходов, вердикт «системно/нет», нет ли рывка.

### B. Dismissible (скрытие карточек/строки)
> Проверь компонент dismissible на `<page>.html`, Chromium, сервер из корня.
> 1) Тап по `[data-dismiss]` в карточке → карточка плавно гаснет (~500ms ease-in-out) и удаляется,
>    нижние ячейки плавно подъезжают (height/padding анимируются к 0, без скачка).
> 2) Удали все карточки по очереди → когда удалена последняя, схлопывается весь `[data-dismiss-row]`.
> 3) Тап по `[data-dismiss="row"]` («Скрыть») → вся строка плавно схлопывается.
> Верни скриншоты середины схлопывания, подтверди отсутствие резкого скачка, вердикт.

### C. Лайк-анимация (lottie)
> Проверь lottie-анимацию «Класс!» на `<page>.html` (обычно lenta-q3.html), Chromium, сервер из корня.
> Тап по кнопке лайка (`.button-klass` / чекбокс) при постановке лайка → поверх кнопки проигрывается
> оверлей `assets/lottie/like.json`, отцентрованный по видимой `.button-container`, и удаляется по завершении.
> На снятие лайка анимация не запускается. Верни скриншот в момент анимации + вердикт, ошибки в консоли.

### D. Произвольная проверка
> Сформулируй конкретный сценарий: страница, последовательность тапов/свайпов, ожидаемый результат
> (визуал + измеримые сигналы: классы, computed style, sessionStorage, URL). Попроси скриншоты до/после
> и чёткий вердикт PASS/FAIL с причиной.
